# План: пересборка `lib/wp-field` с нуля после отката (git reset)

_Создано: 2026-04-06 · формат: execution-ready plan_

> **Связь с существующей документацией:** исторический план de-legacy / bridge остаётся в [`docs/wp-field/plan.md`](wp-field/plan.md). Этот файл — **отдельный дорожный план** для сценария полного отката текущей ветки и повторной реализации требований к демо, ассетам и документации без накопленного технического долга в рабочем дереве.

---

## 1. Goal

Вернуть репозиторий к **чистому известному состоянию** (базовый коммит / upstream), затем **поэтапно** восстановить или заново реализовать:

- демо-страницы **Vanilla (Classic API)** и **Fluent v3** с предсказуемым UI;
- **pipeline ассетов** (UIManager, интеграции, Vite-бандлы) без скрытых зависимостей (в т.ч. jQuery на v3 при необходимости);
- **согласованность** с [`docs/wp-field/supported-matrix.md`](wp-field/supported-matrix.md), [`API.md`](../API.md), [`decision-log.md`](wp-field/decision-log.md).

Итог: **воспроизводимая** среда для интеграторов и **стабильные** админ-страницы демо без фаталов и «сломанной» вёрстки.

---

## 2. Expected Result

- Рабочее дерево соответствует выбранной **базовой точке** (commit/tag); нет незакоммиченных артефактов отменённой линии разработки, если так задумано.
- `composer test`, `composer analyse` (или согласованный CI-набор) проходят на каждом значимом этапе.
- Демо включаются осознанно (**флаг/константа**, не только `WP_DEBUG`), документировано в [`API.md`](../API.md) / README.
- **Vanilla** и **v3** используют **раздельные** стили (SCSS → CSS), без смешения классов.
- **v3**: один предсказуемый JS-бандл для оболочки (без обязательной второй загрузки shared-chunk, если WordPress подключает только entry).
- Документация и `tree.md` отражают фактическую структуру после пересборки.

---

## 3. Context

- В репозитории уже велась работа по демо (Vanilla + React v3, `wp-field-integrations`, отдельные SCSS). Часть правок давала регрессии (fatal из-за отсутствующих fluent-методов, рассинхрон Vite chunks и `wp_enqueue_script`, визуальные артефакты в админке).
- Приоритеты из [`AGENTS.md`](../AGENTS.md): supported matrix, `LegacyWrapperField`, repeater/flexible/UIManager, затем расширение fluent API.
- Источник типов: [`supported-matrix.md`](wp-field/supported-matrix.md) (52 unique + алиасы + `flexible_content` в Fluent).

---

## 4. Constraints

- **Не ломать legacy API** без явной записи в `decision-log.md` и согласования.
- **Не расширять** fluent API «вслепую»; каждый новый fluent-метод — либо паритет с существующим контрактом поля, либо отдельное решение в decision-log.
- Любое изменение **supported behavior** → обновление `API.md` / матрицы / примеров.
- Демо-код проверять по **`Field.php` и реальным классам `Types/*`**, не по README-фантазии.
- После значимых сессий — обновление минимум одного из: `analysis.md`, `plan.md`, `decision-log.md`, `AGENTS.md` (workflow репозитория).

---

## 5. Scope

### 5.1 Allowed (разрешено)

- `git reset` / `git checkout` / отдельная ветка от чистой базы — по вашей процедуре Git.
- Рефакторинг **examples/**, **assets/** (scss, js, vite configs), **legacy/example.php** в рамках демо и документации.
- Добавление/правка **тестов** (Pest) под новые контракты.
- Добавление **отдельных** vite-конфигов под изолированные бандлы (как паттерн «один entry — один артефакт»).
- Правки **`src/Field/Types/*`** для **паритета fluent API** (например `rows()`, `preview()`), если это совпадает с уже используемыми атрибутами в `render()`.

### 5.2 Forbidden (запрещено без отдельного RFC / decision-log)

- Переименование каталога **`legacy/`** или массовое удаление legacy runtime «заодно» с демо.
- Изменение **публичных контрактов** (`FieldInterface`, контейнеры, storage) без обновления `API.md` и тестов.
- Подключение демо **всегда** в продакшене без флага/константы.
- «Тихие» правки **supported-matrix** под фактический код без синхронизации кода и тестов.

---

## 6. Assumptions

- Базовый коммит для отката выбран осознанно (например `origin/version_4` или конкретный SHA); команда владеет историей submodule `woo2iiko`.
- PHP **8.3+** и WordPress **6.0+** сохраняются как целевые версии.
- Сборка фронта (`npm run build` и варианты) доступна в среде разработчика; при отсутствии — зафиксировать в README минимальную команду для CI.

## 7. Execution Plan

### Phase A — Preparation

- [ ] **A1.** Зафиксировать **целевую базу** отката (ветка/tag/commit) и записать её в комментарии к задаче или в `decision-log.md` (одна строка: «база пересборки: …»).
- [ ] **A2.** Сохранить список **требований к результату** (этот документ + ваши устные/чат-требования): Vanilla vs Fluent, отказ от jQuery на v3, матрица типов, GH Pages / статика — что входит в scope.
- [ ] **A3.** Выполнить **git reset** (или новая ветка от чистой базы) по согласованной процедуре; убедиться, что `git status` чистый или осознанно содержит только нужные файлы.
- [ ] **A4.** Прогнать **baseline**: `composer test`, при необходимости `composer analyse`, `npm run build` — зафиксировать, что база зелёная до новых фич.

### Phase B — Implementation (ядро поведения)

- [ ] **B1.** Восстановить или заново ввести **`WP_FIELD_LOAD_DEMOS`** (или эквивалент) в `wp-field.php` с документированным default.
- [ ] **B2.** **Vanilla-демо** (`legacy/example.php` + SCSS): терминология, минимальный шум, таблица аргументов и блоки кода без конфликта с глобальными стилями WP (обёртки, без лишних `!important` без необходимости).
- [ ] **B3.** **Интеграции** (`assets/js/wp-field-integrations.js` + `UIManager`): два пути (classic vs react), зависимости скриптов согласованы с decision-log (color/media/editor без лишнего jQuery в react-пути).
- [ ] **B4.** **Fluent v3-демо** (`examples/v3-demo.php`): только вызовы API, которые **существуют** на классах полей; при необходимости — добавить fluent-методы как тонкие обёртки над `attribute()`.
- [ ] **B5.** **Vite**: отдельная сборка v3 entry в **один** артефакт или явная стратегия загрузки всех chunks WordPress’ом; не полагаться на неподключаемый `client.js`.
- [ ] **B6.** **SCSS демо**: `vanilla-demo.scss` и `v3-demo.scss` → отдельные CSS; `npm run build:scss` в pipeline.

### Phase C — Integration (документация и субмодуль)

- [ ] **C1.** Синхронизировать **README / README.ru / API.md** с двумя демо и числом типов (матрица).
- [ ] **C2.** Обновить **`tree.md`** под новые файлы (vite-конфиги, пути dist).
- [ ] **C3.** При изменении архитектурных решений — **`decision-log.md`**; риски — **`analysis.md`**.
- [ ] **C4.** Если `woo2iiko` подключает submodule: инструкция bump submodule для деплоя.

### Phase D — Validation / Testing

- [ ] **D1.** `composer test` — полный прогон; при изменении типов — точечные unit/feature тесты.
- [ ] **D2.** `composer analyse` / lint по политике проекта.
- [ ] **D3.** `npm run build` (и при необходимости отдельные `build:shell` / `build:wizard`) без ошибок.
- [ ] **D4.** Ручная проверка в админке: **Tools → WP_Field Examples** и **Tools → WP_Field Fluent** — нет fatal, сайдбар/поиск v3 работает, таблицы и код читабельны.
- [ ] **D5.** Контроль **Network**: для v3 нет лишних 404 по JS; размер `v3-demo.js` соответствует ожидаемому полному бандлу (если выбран single-bundle подход).

## 8. Стандарты разработки и тестирования

| Область | Стандарт |
|--------|----------|
| PHP | PSR-совместимый стиль проекта, `composer analyse` / Pint по `composer.json` |
| Тесты | **Pest** (`composer test`, при необходимости `composer test:unit` / `test:feature`) — зелёный прогон перед merge |
| Публичный API | Любое изменение контрактов → **`API.md`** + тесты |
| Документация фактов | **`supported-matrix.md`**, **`decision-log.md`** при архитектурных решениях |
| Frontend | После правок **assets** — `npm run build` (или минимальный набор скриптов из `package.json`) |
| Legacy | Изменения только с осознанием backward compatibility; крупное — запись в decision-log |

- [ ] База отката задокументирована; CI/локальные проверки зелёные.

- [ ] Документация и матрица не противоречат коду.
- [ ] Нет «висячих» обязательных чанков JS для v3 без явной загрузки.
